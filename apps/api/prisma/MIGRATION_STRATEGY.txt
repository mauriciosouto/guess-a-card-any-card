Guess a Card, Any Card — database migration strategy

1) Game-owned PostgreSQL
   - This repo owns `apps/api/prisma/schema.prisma` and all migrations under
     `apps/api/prisma/migrations/`.
   - The admin app is retired for now and does not share this database.
   - Legacy `Puzzle` / `PuzzleStep` (and related gameplay FKs) were removed in favor of
     catalog snapshots on `games` and `user_card_stats.card_id`.

2) Applying migrations
   - Production/staging: `npm run db:deploy` (Prisma `migrate deploy`) with `DATABASE_URL` set.
   - Local iteration: `npm run db:migrate:dev` against a disposable Postgres when experimenting
     with new migrations.
   - Do not use `prisma migrate reset` on shared production data unless you intend to wipe it.

3) First-time setup
   - Copy `.env.example` to `.env` and set `DATABASE_URL`.
   - Run `npm run db:deploy` from the repo (API workspace) to apply pending migrations.

4) Historical note
   - Early migrations may reference `Puzzle` / `PuzzleStep` for shadow DB replay; later migrations
     drop those tables. Fresh databases replay the full chain and end on the catalog-only schema.

5) Supabase Auth vs `users`
   - Align `users.id` with `auth.users.id` when OAuth lands.

6) Seed script
   - Command: `npm run db:seed` (prisma/seed.ts).
   - Today: upserts `Achievement` rows from BLUEPRINT_PRO.md §23.2 (codes FIRST_BLOOD,
     SHARP_EYE, CONSISTENT_SCHOLAR, SET_SPECIALIST, NEVER_GIVE_UP).
   - Idempotent: safe to re-run.

7) Repository / service outline
   - See `DB_LAYER_OUTLINE.txt` for file map (catalog + stats + multiplayer).

8) Production
   - `prisma migrate deploy` against production `DATABASE_URL` as part of deploy.

9) Failed migration (P3018)
   - Mark the failed migration rolled back, fix SQL, redeploy:
     npx prisma migrate resolve --rolled-back "<migration_folder_name>"
     npm run db:deploy
